# Decisions

## 2026-04-25

- Use `SharedPreferences` for the MVP because DataStore was not already available and no new dependencies should be installed without approval.
- Keep navigation as a small in-memory enum because the MVP has only four screens and adding Navigation Compose would add another dependency.
- Use `PackageManager` to list launchable apps and store selected package names locally.
- Include Usage Access permission explanation and settings shortcut, but do not implement usage reading or enforcement in this slice.
- Set `compileSdk` and `targetSdk` to 35 because no installed Android SDK was discoverable in this workspace to derive a local value from.
- Do not pin Gradle's Kotlin toolchain to JDK 17 because this machine exposes JDK 21; keep generated bytecode targeting JVM 17 instead.
- Keep usage limit comparison in a small plain Java class so a temporary dependency-free unit test can run without adding JUnit. Replace this with a standard Android/JUnit test setup when dependency policy allows.
- Use `UsageStatsManager.queryUsageStats` for today's selected-app foreground usage and keep enforcement out of scope.
- Keep the friction flow as an in-app demo route from the dashboard. It prepares countdown and intention state without launching, blocking, or replacing other apps.
- Keep friction state logic in a small plain Java class so it can be tested without adding dependencies.
- Treat the friction screen contract as reusable across Core dashboard demo, future Launcher Mode, and future Overlay Blocker Mode.

## 2026-04-25 Product Direction Update

- Organize the product around three modes: Core Mode, optional Launcher Mode, and optional experimental Overlay Blocker Mode.
- Keep Core Mode as the default path for app selection, usage tracking, daily limits, the reusable friction screen, and a later notification inbox.
- Treat Launcher Mode as a separate optional custom Android launcher. It may randomize or move target app icons only inside Friction Wellbeing's launcher UI and must not attempt to move icons on the user's existing launcher.
- Treat Overlay Blocker Mode as optional and experimental. If implemented later, it will rely on `AccessibilityService` and overlay permission, both of which require clear user disclosure and opt-in.
- Document Overlay Blocker detection as heuristic and fragile. Instagram and YouTube UI changes may break Reels/Shorts detection, and the app must not claim perfect blocking.
- Acknowledge that Overlay Blocker Mode may be risky for Google Play review.
- Current implementation order after the reusable friction screen changed to: harden Overlay Blocker Mode, add optional Launcher Mode, then add notification inbox later.
- Implement Overlay Blocker Mode before Launcher Mode for the first real enforcement slice, but keep it optional, experimental, and local-only.
- Use an `AccessibilityService` for package-level foreground app detection only. The service does not retrieve window content and does not attempt Reels or Shorts detection in this slice.
- Declare `SYSTEM_ALERT_WINDOW` and require the user to grant Display over other apps permission before any overlay is shown.
- Keep Overlay Blocker Mode off by default and store its enablement locally in `SharedPreferences`.
- Add strict overlay mode as an explicit local toggle for phone testing before a target app reaches the daily limit.
- Store only a per-package allow-until timestamp for overlay cooldown. The default allow window is 2 minutes so friction can return quickly during real use.
- Keep overlay eligibility and cooldown rules in a small plain Java class so they can be tested without adding dependencies.
- Use challenge-style friction copy for the overlay instead of a plain intention prompt. Continue requires the countdown to finish and the input to include the expected challenge answer.
- Add overlay repeat modes instead of one fixed cooldown: Light repeats after 10 minutes, Heavy repeats after 2 minutes, and Ultra Focus starts a timed 25-minute target-app lock.
- While Ultra Focus is active, the app disables its overlay mode controls until the timer ends. Users can still use Android system-level controls outside the app.
- Implement icon move-around as a separate Launcher Mode toggle, not as part of Light/Heavy/Ultra Focus.
- Use an Android home intent filter so Friction can be selected as a launcher, while keeping the normal launcher app entry.
- Shuffle only Friction's own launcher grid by storing a local shuffle seed on phone unlock. Do not try to alter icons on the user's existing launcher.
- Split Launcher Mode into Focus Launcher and Shuffle Launcher. Focus Launcher hides selected distraction targets; Shuffle Launcher shows all launchable apps and changes their order after unlock.
- Add Launcher Off as the default launcher profile so selecting Friction as Home does not automatically hide or shuffle icons.
- Add Friction Off as the default overlay mode so overlay friction is explicit opt-in.
- Add Shorts / Reels as a separate overlay friction mode because it needs accessibility window-content retrieval and should not be bundled with package-level selected-app blocking.
- Use the current wallpaper as a best-effort visual background, with a dark fallback when Android does not expose the wallpaper drawable.
- Do not attempt to retrieve icon positions from other launchers because Android does not provide a public API for that layout data.
